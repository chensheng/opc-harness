## Context

当前 `harness-check.js` 脚本在第 4 项检查 "Rust Compilation Check" 中，仅验证 `cargo check` 是否成功退出（exit code 0）。如果编译成功，即使存在警告，也会标记为 PASS 并给满分。

**当前实现：**
```javascript
// 简化的逻辑
const result = execSync('cargo check', { encoding: 'utf-8' });
if (result.includes('Finished')) {
  // PASS - 不考虑警告
}
```

**问题：**
- Rust 编译器会输出警告到 stderr，但不影响 exit code
- 项目可能积累大量警告而不被发现
- 健康评分无法反映代码质量的真实状态

**约束条件：**
- 不能破坏现有的 CI/CD 流程
- 需要向后兼容（允许配置是否启用警告检查）
- 评分算法应该简单透明，易于理解

## Goals / Non-Goals

**Goals:**
1. 在健康评分中考虑 Rust 编译警告数量
2. 提供清晰的警告计数和类型统计
3. 保持评分系统的公平性和可预测性
4. 鼓励开发者及时清理警告

**Non-Goals:**
1. 不实现警告的自动修复（这是 `harness:fix` 的职责）
2. 不区分警告的严重程度（所有警告同等对待）
3. 不改变 Rust 编译器的默认行为
4. 不影响其他检查项的评分逻辑

## Decisions

### Decision 1: 使用正则表达式解析 cargo check 输出

**选择：** 通过解析 `cargo check` 的标准输出来统计警告数量

**理由：**
- `cargo check` 会在输出中包含 "warning:" 字样
- 可以使用简单的正则表达式 `/^warning:/gm` 匹配
- 不需要额外的工具或依赖

**替代方案考虑：**
1. **使用 `cargo clippy`**
   - 优点：更严格的检查
   - 缺点：运行时间长，可能过于严格
   - 决定：保留为未来的增强选项

2. **解析 JSON 输出**
   - 优点：结构化数据，更精确
   - 缺点：需要 `--message-format=json`，增加复杂性
   - 决定：对于当前需求，文本解析足够

### Decision 2: 评分算法 - 每个警告扣 2 分

**选择：** Rust 编译检查的基础分为 12.5 分（总分 100 分的 1/8），每个警告扣 2 分，最低 0 分

**计算公式：**
```
rust_score = max(0, 12.5 - (warning_count * 2))
```

**示例：**
- 0 个警告：12.5 分（满分）
- 3 个警告：6.5 分
- 7 个警告：0 分（扣分上限）

**理由：**
- 12.5 分是 8 个检查项的平均权重
- 每个警告扣 2 分意味着最多容忍 6 个警告
- 超过 6 个警告该项得 0 分，促使开发者清理

**替代方案考虑：**
1. **固定阈值（PASS/FAIL）**
   - 优点：简单
   - 缺点：不够灵活，无法反映渐进改进
   - 决定：采用渐变评分更能激励改进

2. **不同警告类型不同权重**
   - 优点：更精细
   - 缺点：实现复杂，可能引起争议
   - 决定：保持简单，所有警告平等对待

### Decision 3: 输出格式增强

**选择：** 在检查结果中显示警告数量和前 5 个警告摘要

**输出示例：**
```
[4/8] Rust Compilation Check...
  [WARN] Found 3 warnings (-6 points)
  - warning: unused variable `x` at src/main.rs:10
  - warning: unused import `std::fs` at src/lib.rs:5
  - warning: ...
```

**理由：**
- 提供即时的反馈，帮助开发者定位问题
- 限制显示前 5 个避免输出过长
- 明确显示扣分情况

## Risks / Trade-offs

### Risk 1: 误报第三方库的警告

**风险：** `cargo check` 可能会显示依赖库的警告

**缓解：**
- `cargo check` 默认只显示当前 crate 的警告
- 如果需要，可以添加 `--workspace` 参数控制范围
- 当前项目是单一 crate，风险较低

### Risk 2: 评分下降导致 CI 失败

**风险：** 现有项目可能有警告，实施后健康评分下降

**缓解：**
- 在实施前先运行一次检查，了解当前警告数量
- 提供迁移指南，帮助团队清理警告
- 可以考虑设置过渡期（例如：第一个月警告只警告不扣分）

### Risk 3: 警告计数的准确性

**风险：** 正则表达式可能误计数（例如：警告消息中包含 "warning:" 字样）

**缓解：**
- 使用 `^warning:` 锚定行首，减少误报
- 测试多种场景确保准确性
- 如果发现问题，可以改用更精确的解析方法

### Trade-off: 简单性 vs 精确性

**权衡：** 选择简单的文本解析而非复杂的结构化解析

**理由：**
- 当前需求只需要计数，不需要详细信息
- 简单实现更容易维护和理解
- 如果未来需要更多功能，可以升级解析逻辑

## Migration Plan

**部署步骤：**
1. 修改 `scripts/harness-check.js` 添加警告检测逻辑
2. 更新 `openspec/specs/development-workflow/spec.md` 文档
3. 运行 `npm run harness:check` 验证新功能
4. 清理现有的 Rust 警告以达到 100 分

**回滚策略：**
- 如果发现问题，可以临时禁用警告检查（添加配置开关）
- 回滚到之前的版本不会影响项目功能

**验证步骤：**
1. 故意引入一个 Rust 警告
2. 运行 `npm run harness:check`
3. 确认健康评分下降且显示警告信息
4. 清理警告后确认评分恢复
