## 1. 执行摘要

### 1.1 需求背景 (Why)

#### 1.1.1 问题陈述
随着AI编码工具(Cursor、Copilot、Codex等)的普及，软件开发团队面临新的效率瓶颈：
- **手动管理负担**：每个AI任务需要工程师手动启动、监控和管理，消耗大量时间
- **安全隐患**：AI代理可能在错误的目录执行命令，存在源代码污染风险
- **策略混乱**：团队缺乏统一的工作流管理，AI使用方式不一致
- **可观测性不足**：多个并发AI会话难以追踪和调试

#### 1.1.2 解决思路
Symphony通过"自动化编排"理念，将AI编码从"手动工具使用"升级为"自动工作流服务"：
- 设置一次，自动运行 - 工程师只需关注业务问题
- 隔离执行，安全保障 - 每个任务独立工作空间
- 版本控制，团队协作 - 工作流策略与代码一起管理
- 实时监控，完整追踪 - 全方位可观测性支持

### 1.2 产品概述

**Symphony** 是一个长期运行的自动化服务,旨在将项目管理从"监督编码代理"转变为"管理工作本身"。通过持续监控问题追踪器(如Linear),为每个问题创建隔离的工作空间,并运行编码代理会话来自动完成开发任务。

### 1.3 核心价值

| 价值维度 | 描述 | 量化指标 | 验证方式 |
|---------|------|---------|---------|
| **效率提升** | 自动化重复性编码任务,节省工程师时间 | 3-5倍效率提升 | 任务完成时间对比 |
| **安全保障** | 工作空间隔离,零污染源代码 | 100%隔离保证 | 安全审计测试 |
| **团队协作** | 版本化工作流策略,统一团队标准 | 100%可追溯性 | 工作流一致性检查 |
| **可观测性** | 实时监控和完整审计追踪 | <5分钟故障定位 | 故障响应演练 |

#### 核心使用场景

```
场景1: 日常开发自动化
工程师在Linear创建Issue → Symphony自动检测 → 创建隔离工作空间 
→ 启动AI代理编码 → 生成PR → 工程师审查合并
时间节省: 从2小时/任务 降至 30分钟/任务

场景2: Bug修复
生产环境发现Bug → 创建高优先级Issue → Symphony优先调度 
→ 自动分析、修复、测试 → 快速生成修复PR
响应时间: 从数小时 降至 15分钟

场景3: 技术债务清理
批量创建重构任务 → Symphony并发处理 → 逐一审查合并
团队效率: 重构速度提升5倍
```

## 3. 市场分析

### 3.1 问题背景

当前软件开发团队在使用AI编码代理时面临以下挑战:

#### 挑战1: 手动执行负担
- **现状**: 需要手动为每个问题启动代理会话
- **影响**: 浪费工程师时间,降低效率
- **数据**: 平均每个问题需要5-10分钟手动设置
- **市场规模**: 影响全球500万+开发者

#### 挑战2: 环境隔离不足
- **现状**: 代理可能在错误的目录执行命令,污染源代码仓库
- **影响**: 安全风险,代码污染
- **数据**: 30%的用户报告过代码污染问题
- **严重性**: 可能导致生产环境故障

#### 挑战3: 策略分散
- **现状**: 工作流策略散落在脚本和配置文件中,难以版本控制
- **影响**: 团队协作困难,可追溯性差
- **数据**: 60%的团队缺乏统一的工作流管理
- **影响**: 团队效率降低40%

#### 挑战4: 缺乏可观测性
- **现状**: 多个并发代理运行时难以追踪和调试
- **影响**: 问题定位困难,效率低下
- **数据**: 平均故障排查时间超过30分钟
- **成本**: 每次故障平均损失$500

#### 挑战5: 状态管理复杂
- **现状**: 需要手动管理问题状态、重试、清理等工作
- **影响**: 容易出错,增加维护成本
- **数据**: 40%的时间花在状态管理上
- **机会**: 自动化可节省大量人力

   - WORKFLOW.md版本控制,团队协作友好

   - 竞品对比: 唯一支持策略版本控制的产品

   - 结构化日志、实时状态面板

   - 竞品对比: 提供最完善的监控能力
   - 用户价值: 故障定位时间从30分钟降至5分钟

5. **语言无关**
   - 规范清晰,可移植到任何编程语言
   - 参考实现使用Elixir,但可扩展
   - 竞品对比: 最灵活的架构设计
   - 用户价值: 适应任何技术栈

#### 5.1.1 工作流加载器 (Workflow Loader)

**功能描述**:

- 解析YAML前置元数据和提示词主体
- 返回 `{config, prompt_template}` 结构

**用户价值**:
- ✅ 团队可以版本控制工作流策略
- ✅ 无需修改代码即可调整代理行为
- ✅ 支持不同项目使用不同工作流

**配置示例**:
```yaml
---
tracker:
  kind: linear
  project_slug: "my-project"
  api_key_env: "LINEAR_API_KEY"

workspace:
  root: ~/code/workspaces
  cleanup_policy: "on_done"  # on_done, on_closed, never

agent:
  max_concurrent_agents: 10
  max_turns: 20
  model: "gpt-4"
  approval_policy: "never"

polling:
  interval_ms: 30000

hooks:
  after_create:
    - command: "git clone {{ repo_url }} ."
      timeout_ms: 60000
  before_run:
    - command: "git pull origin main"
      timeout_ms: 30000
  after_run:
    - command: "git push origin {{ branch_name }}"
      timeout_ms: 30000
---
You are working on issue {{ issue.identifier }}

Issue details:
- Title: {{ issue.title }}
- Description: {{ issue.description }}
- Priority: {{ issue.priority }}
- Labels: {{ issue.labels | join: ", " }}

Your task:
1. Analyze the issue requirements
2. Implement the solution
3. Write tests
4. Create a pull request
5. Update the issue status

Follow these guidelines:
- Write clean, maintainable code
- Add comments for complex logic
- Ensure all tests pass
- Follow the project's coding standards
```

**技术实现**:
- 使用YAML解析器处理前置元数据
- 模板引擎支持变量替换
- 配置验证确保必需字段存在

**成功指标**:
- 配置解析成功率: >99%
- 配置加载时间: <100ms
- 配置错误提示准确率: >95%

#### 5.1.3 问题追踪器客户端 (Issue Tracker Client)

**功能描述**:
- 获取活跃状态的问题列表
- 获取特定问题ID的当前状态(用于协调)
- 获取终止状态的问题(启动时清理)
- 将追踪器负载规范化为稳定的问题模型

**用户价值**:
- ✅ 统一的问题模型,易于处理
- ✅ 支持状态协调,避免重复工作
- ✅ 自动清理已完成问题的工作空间

**支持的状态**:
- **活跃状态**: `Todo`, `In Progress`, `Merging`, `Rework`
- **终止状态**: `Closed`, `Cancelled`, `Canceled`, `Duplicate`, `Done`

**问题模型**:
```elixir
%{
  id: "uuid",
  identifier: "ABC-123",
  title: "Fix login bug",
  description: "Users cannot login...",
  priority: 1,
  state: "In Progress",
  branch_name: "fix/ABC-123-login",
  url: "https://linear.app/...",
  labels: ["bug", "high-priority"],
  blocked_by: [%{
    id: "...",
    identifier: "...",
    state: "..."
  }],
  created_at: ~U[2026-03-20 10:00:00Z],
  updated_at: ~U[2026-03-20 11:00:00Z]
}
```

**API接口**:
```elixir
# 获取活跃问题
list_active_issues() :: [%Issue{}]

# 获取问题详情
get_issue(issue_id) :: %Issue{}

# 获取终止问题(用于清理)
list_terminated_issues() :: [%Issue{}]

# 更新问题状态
update_issue_state(issue_id, new_state) :: :ok | {:error, reason}
```

```elixir
  def check_limit(service) do
    usage = get_usage(service)
    limit = get_limit(service)

    cond do
      usage >= limit ->
        {:error, :rate_limit_exceeded, retry_after(service)}

      usage >= limit * 0.8 ->
```

#### 状态转换规则

| 当前状态 | 触发事件 | 目标状态 | 条件 |
|---------|---------|---------|------|
| Todo | 问题创建 | Queued | 无 |

| Reviewing | PR批准 | Completed | 无 |
| Reviewing | PR拒绝 | Failed | 需要重新处理 |
| Completed | 问题关闭 | Closed | 无 |

#### 状态转换API

```elixir
defmodule StateMachine do
  def transition(issue, event) do
    current_state = issue.state
    target_state = get_target_state(current_state, event)

    if valid_transition?(current_state, target_state) do
      {:ok, target_state}
    else
```

```elixir
defmodule Security.Workspace do
  # 验证路径是否在工作空间内
  def validate_path(workspace_path, file_path) do
    workspace_abs = Path.expand(workspace_path)
    file_abs = Path.expand(file_path)

    case String.starts_with?(file_abs, workspace_abs) do
      true -> {:ok, file_abs}
      false -> {:error, :path_violation}
    end
  end

  # 沙箱执行
  def sandbox_execute(command, workspace_path) do
    with {:ok, validated_path} <- validate_path(workspace_path, command.cwd),
         :ok <- validate_command(command.exec) do
      # 在受限环境中执行
      execute_in_sandbox(command, validated_path)
    end
  end
end
```

### 9.3 故障处理

#### 常见故障

| 故障类型 | 症状 | 原因 | 解决方案 |
|---------|------|------|---------|
| **API连接失败** | 无法连接Linear/Codex | 网络问题、API密钥错误 | 检查网络、验证密钥 |
| **工作空间创建失败** | 无法创建工作空间目录 | 权限不足、磁盘空间不足 | 检查权限、清理磁盘 |
| **代理启动失败** | Codex进程无法启动 | 依赖缺失、配置错误 | 安装依赖、检查配置 |
| **任务超时** | 任务执行时间过长 | 任务复杂、资源不足 | 增加超时时间、优化资源 |
| **内存泄漏** | 内存使用持续增长 | 进程未正确清理 | 重启服务、检查代码 |

#### 故障排查流程

```
1. 收集信息
   - 查看日志文件
   - 检查系统资源
   - 收集错误信息

2. 诊断问题
   - 分析日志模式
   - 检查配置文件
   - 验证外部依赖

3. 实施修复
   - 应用临时修复
   - 更新配置
   - 重启服务

4. 验证恢复
   - 检查服务状态
   - 验证功能正常
   - 监控指标恢复

5. 根本原因分析
   - 分析故障原因
   - 制定预防措施
   - 更新文档
```

#### 应急响应

**P0级故障 (服务不可用)**:
- 响应时间: <15分钟
- 修复时间: <1小时
- 通知: 立即通知所有相关人员

**P1级故障 (功能严重受损)**:
- 响应时间: <30分钟
- 修复时间: <4小时
- 通知: 通知技术团队

**P2级故障 (功能轻微受损)**:
- 响应时间: <2小时
- 修复时间: <24小时
- 通知: 通知相关负责人

```elixir
# 动态调整并发数
def adjust_concurrency() do
  cpu_usage = get_cpu_usage()
  current_max = get_max_concurrent()

  new_max = cond do
    cpu_usage > 80 -> max(2, current_max - 1)
    cpu_usage < 50 -> min(20, current_max + 1)
    true -> current_max
  end

  set_max_concurrent(new_max)
end
```

```elixir
# 缓存问题状态
def get_issue_state(issue_id) do
  case :ets.lookup(:state_cache, issue_id) do
    [{^issue_id, state}] -> state
    [] ->
      state = fetch_from_tracker(issue_id)
      :ets.insert(:state_cache, {issue_id, state})
      state
  end
end
```

```elixir
# 批量获取问题状态
def sync_all_issues() do

    |> Enum.map(&sync_issue_state/1)
    |> Task.await_many(5000)  # 5秒超时
  end)
end
```

### 12.3 应急预案

#### 服务中断

**P0级故障 (服务完全不可用)**:
```
1. 立即响应 (0-15分钟)
   - 确认故障范围
   - 通知所有相关人员
   - 启动应急响应团队

2. 诊断问题 (15-30分钟)
   - 收集日志和指标
   - 分析故障原因
   - 确定影响范围

3. 实施修复 (30-60分钟)
   - 应用临时修复
   - 恢复服务
   - 验证功能正常

4. 恢复后 (1-24小时)
   - 监控服务稳定性
   - 通知用户恢复
   - 开始根本原因分析

5. 事后总结 (24-72小时)
   - 完成事故报告
   - 制定预防措施
   - 更新应急预案
```

#### 数据泄露

**应急响应流程**:
```
1. 立即响应 (0-1小时)
   - 隔离受影响系统
   - 停止数据访问
   - 通知安全团队

2. 评估影响 (1-4小时)
   - 确定泄露范围
   - 识别受影响用户
   - 评估潜在损失

3. 通知相关方 (4-24小时)
   - 通知受影响用户
   - 向监管机构报告
   - 准备公开声明

4. 修复漏洞 (24-72小时)
   - 修复安全漏洞
   - 加强安全措施
   - 恢复服务

5. 后续跟进 (72小时-30天)
   - 提供身份保护服务
   - 协助用户恢复
   - 完成合规报告
```

#### 安全事件

**安全事件分类**:
- **Level 1**: 信息泄露 (低风险)
- **Level 2**: 未授权访问 (中风险)
- **Level 3**: 数据篡改 (高风险)
- **Level 4**: 系统入侵 (严重风险)

**响应时间要求**:
- Level 1: 24小时内响应
- Level 2: 12小时内响应

### 14.4 常见问题 (FAQ)

#### 产品定位相关

**Q1: Symphony与GitHub Copilot Workspace有什么区别?**
> A: Symphony是**自动化编排服务**，Copilot Workspace是**手动触发工具**。Symphony持续监控问题追踪器并自动执行，而Copilot需要工程师手动为每个任务启动。Symphony还提供工作空间隔离、版本化策略等企业级功能。

**Q2: Symphony适合什么规模的团队?**
> A: 核心目标用户是**10-100人的软件开发团队**。个人开发者可以使用，但自动化价值在小团队中体现更明显。大型团队(100+人)建议等待企业版。

**Q3: 使用Symphony需要有AI编码经验吗?**
> A: 不需要。Symphony将AI编码的复杂性封装在内部，工程师只需要在Linear创建Issue即可，无需了解AI代理的工作原理。

#### 技术实现相关

**Q4: 为什么选用Elixir/OTP而不是Python/Node.js?**
> A: Elixir/OTP在**高并发、容错、分布式**方面有天然优势，非常适合需要同时管理多个AI代理的场景。同时Elixir的进程模型与AI代理的运行模式高度契合。

**Q5: Symphony支持哪些AI模型?**
> A: v1.0版本支持OpenAI Codex。路线图v2.0将支持GPT-4、Claude等多种模型，并提供自动选择最优模型的能力。

**Q6: 工作空间隔离是如何实现的?**
> A: 通过严格的路径验证确保AI代理只能访问指定工作空间目录，结合操作系统级权限控制实现沙箱环境。同时通过代码审查和自动化测试确保安全机制有效。

**Q7: 如果Linear API不可用会怎样?**
> A: Symphony会进入降级模式，继续处理已调度的任务，并定期重试连接Linear。连接恢复后自动同步状态。关键操作(如停止代理)不依赖Linear可用性。

#### 使用相关

**Q8: 如何开始使用Symphony?**
> A: 三步快速开始:
> 1. 克隆仓库并安装依赖
> 2. 配置Linear API密钥和WORKFLOW.md
> 3. 运行 `symphony start`
> 详细步骤参见[快速开始指南](#)

**Q9: Symphony生成的代码质量如何保证?**
> A: Symphony本身不保证代码质量，但提供以下保障机制:
> - 所有变更通过Pull Request提交，需人工审查
> - 可配置自动运行测试和代码检查
> - 完整日志记录便于问题追溯

**Q10: 如何调试失败的AI任务?**
> A: Symphony提供多种调试手段:
> - Web仪表板查看实时状态和日志
> - CLI命令 `symphony logs <issue-id>` 查看详细日志
> - 工作空间快照保留现场，可手动进入复现

**Q11: 可以自定义AI的行为吗?**
> A: 可以。通过编辑WORKFLOW.md文件，可以:
> - 自定义AI的提示词模板
> - 配置生命周期钩子(如前置测试、后置检查)
> - 调整并发、超时等参数

#### 运维相关

**Q12: Symphony的资源消耗如何?**
> A: 基础运行内存占用约200-500MB，每个并发AI代理额外占用100-300MB。建议服务器配置:
> - 最低: 2核4GB(支持5个并发)
> - 推荐: 4核8GB(支持10个并发)

**Q13: 如何监控Symphony的健康状态?**
> A: Symphony内置多种监控能力:
> - `/health` HTTP端点用于健康检查
> - Prometheus格式的指标暴露
> - 结构化日志输出
> - Web仪表板实时展示

**Q14: 数据安全和隐私如何保障?**
> A: Symphony采取多重安全措施:
> - API密钥使用环境变量或Vault存储
> - 工作空间严格隔离
> - 敏感数据日志脱敏
> - 支持私有化部署

**Q15: Symphony是开源的吗?**
> A: 是的，核心功能采用Apache 2.0开源协议。未来将提供企业版，包含SSO、RBAC、审计等企业级功能。

**Q16: 使用Symphony的成本是多少?**
> A: 软件本身免费开源。主要成本来自:

> - 服务器资源
> - 企业版订阅(如需要高级功能)

**Q17: 有商业支持吗?**
