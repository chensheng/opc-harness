## 1. PRD 综述

### 1.1 一句话简介

Coding Harness 是一个基于 Claude AI 的**自主编码代理编排系统**，采用"人在回路"(Human-in-the-Loop)设计理念，通过里程碑驱动的多会话开发工作流，实现长时间、复杂任务的自动化编码。

### 1.2 核心价值主张

| 为谁          | 解决什么问题               | 达成什么结果                     |
| ------------- | -------------------------- | -------------------------------- |
| 独立开发者    | 时间有限，需要快速验证想法 | 3-5天工作量压缩到4小时内完成     |
| 技术团队      | 重复性工作多，协作成本高   | 统一编码规范，减少代码审查时间   |
| DevOps 工程师 | 手动配置繁琐，容易出错     | 自动化测试和部署，减少人为错误   |
| 开源维护者    | Issue 积压，响应不及时     | 自动化处理 Issue，提升社区活跃度 |

### 1.3 产品愿景

> **成为开发者最信任的 AI 编码伙伴，让每个人都能快速构建高质量软件**

### 1.4 关键差异化

| 维度        | Coding Harness | 竞品 A (Factory) | 竞品 B (Archon) |
| ----------- | -------------- | ---------------- | --------------- |
| 开源免费    | ✅             | ❌ 商业产品      | ✅              |
| HITL 检查点 | ✅ 8个         | ❌ 无            | ✅ 3个          |
| GitLab 集成 | ✅             | ❌               | ✅              |
| 本地模式    | ✅             | ❌               | ❌              |
| 多规格并行  | ✅             | ❌               | ❌              |
| 学习成本    | 🟢 低          | 🟡 中            | 🔴 高           |
| 部署复杂度  | 🟢 低          | 🟡 中            | 🔴 高           |

## 2. 背景与机会

### 2.1 问题定义

#### 当前开发者的三大痛点

**🔴 痛点 1: AI 编码工具的信任危机**

> "我使用过多个 AI 编码工具，但总是担心它们生成的代码质量。没有人工审查，我不敢直接部署到生产环境。"  
> —— 某科技公司技术负责人

- **数据**: 70% 的开发者担心 AI 生成代码的质量
- **影响**: 限制了 AI 工具在生产环境的应用
- **机会**: HITL 检查点机制可以建立信任

**🔴 痛点 2: 上下文限制是最大障碍**

> "我的项目有 10 万行代码，AI 工具无法理解整个项目上下文，只能处理小片段。"  
> —— 独立开发者

- **数据**: 85% 的开发者认为上下文窗口限制是主要障碍
- **影响**: 无法处理复杂、长时间任务
- **机会**: 多会话编排突破上下文限制

**🟡 痛点 3: 与现有工作流集成困难**

> "我们团队使用 GitLab 进行代码审查，AI 工具必须能无缝集成到这个流程中。"  
> —— 技术团队负责人

- **数据**: 80% 的企业使用 GitLab/GitHub
- **影响**: 工具孤岛增加学习成本
- **机会**: GitLab 集成 + 文件模式双模式支持

**详细需求**:

| 需求 ID  | 需求描述           | 验收标准                                      | 优先级 |

| FR-002-2 | Spec 转 Issues     | 自动将 Spec 分解为 GitLab Issues              | P0     |
| FR-002-3 | Issue 状态追踪     | 实时同步 Issue 状态 (open/in-progress/closed) | P0     |

##### FR-004: HITL (Human-in-the-Loop) 检查点

**功能描述**: 8 个关键决策点的人工审查机制，确保代码质量。

**用户场景**: 用户希望在关键决策点保持控制权，确保 AI 输出质量。

**检查点列表**:

| 检查点 ID | 名称                 | 触发时机               | 审查内容                     | 预估耗时 | 可自动接受 |
| --------- | -------------------- | ---------------------- | ---------------------------- | -------- | ---------- |
| CP-001    | project_verification | Initializer 开始前     | 项目目录验证、Git 仓库检查   | < 1min   | ✅         |
| CP-002    | spec_to_issues       | Initializer 分解任务后 | Issue 列表质量、优先级合理性 | 2-5min   | ✅         |
| CP-003    | issue_enrichment     | Issue 丰富化后         | 上下文信息完整性             | 1-2min   | ✅         |
| CP-004    | regression_approval  | 每次编码会话前         | 回归测试结果审查             | 1-3min   | ✅         |
| CP-005    | issue_selection      | 选择下一个 Issue 时    | Issue 优先级确认             | < 1min   | ✅         |
| CP-006    | issue_closure        | Issue 完成后           | 实现完整性、测试覆盖率       | 2-5min   | ❌         |
| CP-007    | mr_phase_transition  | 所有 Issue 完成后      | 是否进入 MR 创建阶段         | 1-2min   | ✅         |
| CP-008    | mr_review            | MR 创建后              | MR 描述、变更内容审查        | 3-10min  | ❌         |

**自动接受模式**:

- 支持全局自动接受 (跳过所有检查点)
- 支持按检查点类型配置自动接受
- 适合信任度高的场景或批量运行

##### FR-005: 质量门禁系统

**功能描述**: 多层次质量检查机制，防止错误累积。

**质量门禁列表**:

| 门禁 ID | 检查项   | 命令                     | 失败处理          | 优先级 |
| ------- | -------- | ------------------------ | ----------------- | ------ |
| QG-001  | 代码检查 | `ruff check`             | 最多 3 次修复尝试 | P0     |
| QG-002  | 类型检查 | `pyright`                | 最多 3 次修复尝试 | P0     |
| QG-003  | 单元测试 | `pytest`                 | 最多 3 次修复尝试 | P0     |
| QG-004  | 回归测试 | 已完成功能的回归检查     | 失败后暂停并通知  | P1     |
| QG-005  | E2E 测试 | `puppeteer` 浏览器自动化 | 失败后暂停并通知  | P1     |

**失败处理策略**:

- 最多 3 次修复尝试
- 超过限制后跳过并记录警告
- Git 回滚机制保护代码库

##### FR-006: 守护进程架构 (Daemon)

**功能描述**: 后台守护进程管理代理生命周期，TUI 可独立重启。

**核心特性**:

- Unix Socket 通信
- 代理状态持久化
- 日志文件管理
- TUI 与代理解耦

**架构优势**:

- TUI 崩溃不影响运行中的代理
- 支持远程监控
- 多 TUI 实例连接同一守护进程

**部署架构**:

```
┌─────────────────┐
│   TUI 进程      │
└────────┬────────┘
         │ Unix Socket
┌────────▼────────┐
│  守护进程       │
│  (Daemon)       │
└────────┬────────┘
         │
    ┌────┴────┬─────────┬─────────┐
    ↓         ↓         ↓         ↓
┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐
│Agent1│ │Agent2│ │Agent3│ │ ...  │
└──────┘ └──────┘ └──────┘ └──────┘
```

##### FR-007: 多规格并行 (Multi-Spec)

**功能描述**: 同时处理多个 Spec 文件，每个 Spec 独立工作空间。

**使用场景**:

- 多功能并行开发
- 不同项目同时处理
- A/B 测试不同实现方案

**隔离机制**:

- 独立的 agent_dir (`.claude-agent/{spec_slug}-{spec_hash}`)

- 独立的状态追踪

##### FR-008: 可插拔代码质量系统

**功能描述**: 支持目标项目使用任意语言和工具链。

**配置示例**:

```json
{

    "lint": "npm run lint",
    "type_check": "npm run type-check",
    "test": "npm test"
  }
}
```

**支持的语言**:

- Python (Ruff, Pyright)
- JavaScript/TypeScript (ESLint, TypeScript)
- Go (golint, go vet)
- Rust (clippy, cargo check)
- Java (Checkstyle, SpotBugs)

##### FR-009: 上下文丰富化 (Issue Enrichment)

**功能描述**: 使用外部 API 增强 Issue 上下文信息。

**集成服务**:

- **Context7**: 库文档搜索
- **SearXNG**: Web 搜索
- **本地代码库**: Grep 搜索、文件结构分析

**触发条件**:

- LLM 判断需要额外上下文
- 用户配置强制丰富化

##### FR-010: 会话交接机制

**功能描述**: 多会话之间的结构化信息传递。

**交接内容**:

### Next Steps

### 11.4 常见问题 (FAQ)

**Q1: Coding Harness 与其他 AI 编码工具有什么区别?**  
**A**: Coding Harness 的核心优势在于: 1) 多会话编排突破上下文限制 2) 8 个 HITL 检查点确保代码质量 3) 双模式支持 (GitLab + 本地文件) 4) 开源免费降低使用门槛

**Q2: 如何选择 GitLab 模式和文件模式?**  
**A**: GitLab 模式适合团队协作、需要代码审查的场景；文件模式适合个人项目、快速原型验证的场景。

**Q3: 检查点可以跳过吗?**  
**A**: 可以。支持全局自动接受模式，也可以按检查点类型配置自动接受。

**Q4: Coding Harness 支持哪些编程语言?**  
**A**: 理论上支持所有编程语言。通过可插拔代码质量系统，可以适配不同项目的工具链。

**Q5: 如何保证 AI 生成代码的质量?**  
**A**: 通过多层次质量保障机制: 8 个 HITL 检查点、质量门禁系统、回归测试、代码审查、Git 回滚机制。
