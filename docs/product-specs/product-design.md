## 1. PRD 综述

### 1.1 一句话简介

OPC-HARNESS 是一个**AI驱动的一人公司操作系统**，为独立创造者整合产品构思(Vibe Design)、快速构建(Vibe Coding)、增长运营(Vibe Marketing)三大模块，实现从想法到产品的全流程闭环。

### 1.2 核心价值主张

| 为谁                  | 解决什么问题               | 达成什么结果                     |
| --------------------- | -------------------------- | -------------------------------- |
| 独立创造者/自由职业者 | 一人创业流程割裂、工具分散 | 一站式完成产品从0到1的全流程     |
| 技术型创业者          | 有技术但不懂产品设计和营销 | AI辅助完成产品设计和增长运营     |
| 设计师/产品经理       | 有创意但缺乏技术实现能力   | AI辅助快速构建MVP原型            |
| 内容创作者            | 需要快速发布工具和变现产品 | 快速创建个人工具，实现多元化收入 |

### 1.3 产品愿景

> **成为独立创造者最信赖的AI创业伙伴，让每个人都能成为一人公司**

### 1.4 关键差异化

| 维度          | OPC-HARNESS  | 竞品 A (Bubble) | 竞品 B (Framer) | 竞品 C (Softr) |
| ------------- | ------------ | --------------- | --------------- | -------------- |
| AI 驱动全流程 | ✅ 三位一体  | ❌ 仅搭建       | ❌ 仅设计       | ❌ 仅搭建      |
| Vibe 方法论   | ✅ 系统化    | ❌ 无           | ❌ 无           | ❌ 无          |
| 一键部署      | ✅           | ⚠️ 部分         | ⚠️ 部分         | ✅             |
| 增长运营集成  | ✅           | ❌              | ❌              | ❌             |
| 本地化部署    | ✅ 支持      | ❌ 云端         | ❌ 云端         | ❌ 云端        |
| 学习成本      | 🟢 低        | 🟡 中           | 🟢 低           | 🟢 低          |
| 定价模式      | 🟢 免费/开源 | 🔴 订阅制       | 🔴 订阅制       | 🔴 订阅制      |

## 2. 背景与机会

### 2.1 问题定义

#### 当前独立创造者的四大痛点

**🔴 痛点 1: 工具分散，流程割裂**

> "我使用Figma设计产品，用VS Code写代码，用Notion管理文档，用ConvertKit发邮件，用Google Analytics看数据... 每天都在不同工具间切换，效率极低。"  
> —— 某独立开发者

- **数据**: 70% 的独立创造者每天使用5+工具
- **影响**: 上下文切换成本高，难以形成工作流
- **机会**: 一站式整合平台

**🔴 痛点 2: 技能短板，单打独斗困难**

> "我是一名开发者，但我不懂UI设计，也不懂营销推广。想做一个产品，但总是卡在某些环节。"  
> —— 某技术创业者

- **数据**: 85% 的独立创造者认为跨领域技能是最大障碍
- **影响**: 项目难以完成，MVP开发周期长
- **机会**: AI辅助补齐技能短板

**🟡 痛点 3: 从想法到产品的路径不清晰**

> "我有很多想法，但不知道怎么验证，不知道怎么搭建，更不知道怎么推广。"  
> —— 某内容创作者

- **数据**: 90% 的创意项目止步于构思阶段
- **影响**: 创新想法无法转化为实际价值
- **机会**: 结构化方法论引导

**🟡 痛点 4: 缺乏数据驱动的增长运营**

> "产品上线了，但不知道怎么获取用户，不知道怎么优化转化，也不知道哪个渠道有效。"  
> —— 某SaaS创业者

- **数据**: 80% 的独立产品缺乏系统的增长策略
- **影响**: 产品难以获得用户，收入不稳定
- **机会**: 集成增长运营工具和方法

### 4.2 用户故事

#### 用户故事 1: 产品构思 (Vibe Design)

**作为** 一名独立开发者，  
**我希望** 通过自然语言描述我的想法，AI能帮我完善产品构思，  
**以便** 快速验证想法的可行性。

**验收标准**:

- [ ] 能够用几句话描述产品想法
- [ ] AI能生成产品需求文档 (PRD)
- [ ] AI能生成用户画像和用户故事
- [ ] AI能提供竞品分析和差异化建议
- [ ] AI能评估技术可行性和开发工作量

#### 用户故事 2: 快速构建 (Vibe Coding)

**作为** 一名设计师，  
**我希望** 只需描述产品需求，AI 就能自动完成所有编码工作，  
**以便** 无需任何编程知识就能获得可运行的产品。

**验收标准**:

- [ ] AI 能根据 PRD 自动生成完整的项目代码
- [ ] 支持多技术栈自动选择 (React/Vue/Next.js 等)
- [ ] 自动生成测试用例并运行验证
- [ ] 支持一键部署到云端
- [ ] 提供代码审查和迭代优化能力
- [ ] 能够处理复杂的业务逻辑和数据库设计

#### 用户故事 3: 增长运营 (Vibe Marketing)

**作为** 一名内容创作者，  
**我希望** AI能帮我制定增长策略和推广计划，  
**以便** 让产品获得更多用户。

**验收标准**:

- [ ] AI能生成产品发布策略
- [ ] AI能生成社交媒体推广文案
- [ ] 支持多渠道发布(Product Hunt, Twitter等)
- [ ] 提供数据分析和转化优化建议
- [ ] 提供用户反馈收集和分析功能

### 6.2 Vibe Coding 核心机制

#### 6.2.1 多会话编排架构

Vibe Coding 采用**多会话编排**(Multi-Session Orchestration) 架构，将复杂的编码任务分解为多个可管理的子任务，由不同的 AI代理协同完成。

```
┌─────────────────────────────────────────────────────────────┐
│                    Vibe Coding 架构                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  📋 PRD 文档                                                │
│       ↓                                                     │
│  ┌─────────────────┐                                       │
│  │ Initializer     │ 环境初始化、任务分解                   │
│  │ Agent           │ → 创建 Milestone                      │
│  │                 │ → 分解为 Issues                       │
│  └────────┬────────┘                                       │
│           ↓                                                 │
│  ┌─────────────────┐                                       │
│  │ Coding Agent 1  │ Issue #1: 用户认证                     │
│  │ Coding Agent 2  │ Issue #2: 登录页面                     │
│  │ Coding Agent 3  │ Issue #3: 数据库设计                   │
│  │ ...             │ (并行执行)                            │
│  └────────┬────────┘                                       │
│           ↓                                                 │
│  ┌─────────────────┐                                       │
│  │ MR Creation     │ 汇总所有提交，创建合并请求              │
│  │ Agent           │ → 生成 MR 描述                         │
│  │                 │ → 关联 Issues                         │
│  └─────────────────┘                                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 6.2.2 HITL 检查点机制

在关键决策点设置人工审查环节，确保代码质量和方向正确：

| 检查点     | 触发时机           | 审查内容                     | 预计耗时 |
| ---------- | ------------------ | ---------------------------- | -------- |
| **CP-001** | Initializer 开始前 | 项目目录验证、Git 仓库检查   | < 1min   |
| **CP-002** | 任务分解后         | Issue 列表质量、优先级合理性 | 2-5min   |
| **CP-003** | Issue 丰富化后     | 上下文信息完整性             | 1-2min   |
| **CP-004** | 编码会话前         | 回归测试结果审查             | 1-3min   |
| **CP-005** | 选择下一个 Issue   | Issue 优先级确认             | < 1min   |
| **CP-006** | Issue 完成后       | 实现完整性、测试覆盖率       | 2-5min   |
| **CP-007** | 所有 Issue 完成后  | 是否进入 MR 创建阶段         | 1-2min   |
| **CP-008** | MR 创建后          | MR 描述、变更内容审查        | 3-10min  |

**自动接受模式**:

- 支持配置自动接受信任度高的检查点
- 适合批量运行和成熟项目
- 用户可随时切换手动审查模式

#### 6.2.3 质量门禁系统

多层次质量保障机制，确保生成的代码符合生产标准：

| 门禁       | 检查项   | 命令示例                  | 失败处理          |
| ---------- | -------- | ------------------------- | ----------------- |
| **QG-001** | 代码检查 | `ruff check` / `eslint .` | 最多 3 次修复尝试 |
| **QG-002** | 类型检查 | `pyright` / `tsc`         | 最多 3 次修复尝试 |
| **QG-003** | 单元测试 | `pytest` / `npm test`     | 最多 3 次修复尝试 |
| **QG-004** | 回归测试 | 已完成功能验证            | 失败后暂停并通知  |
| **QG-005** | E2E 测试 | `puppeteer` 浏览器自动化  | 失败后暂停并通知  |

**失败恢复策略**:

- Git 回滚机制保护代码库
- 自动分析失败原因并尝试修复
- 超过限制后跳过并记录警告
- 用户可手动介入调整

#### 6.2.4 守护进程架构

后台守护进程管理 AI代理生命周期，确保稳定性和可靠性：

```
┌─────────────────┐
│   Web 界面      │ ← 用户交互层
└────────┬────────┘
         │ HTTP/WebSocket
┌────────▼────────┐
│  守护进程       │ ← 核心编排层
│  (Daemon)       │   - 代理状态管理
│                 │   - 任务调度
│                 │   - 日志收集
└────────┬────────┘
         │
    ┌────┴────┬─────────┬─────────┐
    ↓         ↓         ↓         ↓
┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐
│Agent1│ │Agent2│ │Agent3│ │ ...  │
└──────┘ └──────┘ └──────┘ └──────┘
  AI编码代理集群
```

**核心优势**:

- TUI/界面崩溃不影响运行中的代理
- 支持断点续传和状态恢复
- 日志文件持久化存储
- 支持远程监控和管理

```
│ 快捷键：s=启动/停止 p=暂停 l=查看日志 r=审查检查点 ?=帮助     │
└─────────────────────────────────────────────────────────────┘
```

**设计原则**:

- **透明化**: 实时显示 AI 的工作进展和当前状态
- **可干预**: 关键检查点需要人工审查和批准
- **易监控**: 提供详细日志和进度追踪
- **低干扰**: 自动化执行，仅在必要时请求用户输入

### 7.3 交互原则

| 原则           | 说明               | 实现                 |
| -------------- | ------------------ | -------------------- |
| **即时反馈**   | 用户操作后立即响应 | 加载动画、进度条     |
| **渐进式披露** | 复杂功能逐步展示   | 向导式流程、折叠面板 |
| **智能默认**   | 提供合理的默认选项 | AI自动生成、模板选择 |
| **可撤销操作** | 关键操作可撤销     | 版本历史、撤销按钮   |
| **上下文帮助** | 需要时提供帮助     | 工具提示、AI建议     |
| **一致性**     | 界面和交互保持一致 | 统一设计语言         |

### 10.2 应急预案

#### 场景 1: AI 服务中断

**触发条件**: AI API 连续 1 小时无法访问

**应急措施**:

1. 自动切换到备用 AI 模型
2. 通知用户服务降级
3. 暂停新任务创建
4. 保存当前用户数据
5. 持续监控服务恢复
6. 恢复后验证数据完整性

#### 场景 2: 严重安全漏洞

**触发条件**: 发现 CVSS > 7.0 的安全漏洞

**应急措施**:

1. 立即启动应急响应
2. 评估漏洞影响范围
3. 开发安全补丁
4. 通知受影响用户
5. 发布安全更新
6. 事后复盘改进

#### **场景 3: AI代理执行错误**

**触发条件**: AI代理在错误的目录执行命令或生成有害代码

**应急措施**:

1. 立即停止所有运行中的代理
2. Git 自动回滚到最近的安全提交点
3. 分析错误日志，定位问题根因
4. 检查工作空间隔离机制是否失效
5. 调整提示词和约束条件
6. 恢复运行时增加额外的检查点审查
7. 通知受影响的用户

#### **场景 4: 检查点大量拒绝**

**触发条件**: 超过 30% 的检查点被用户拒绝

**应急措施**:

1. 暂停自动接受模式
2. 收集用户拒绝原因和反馈
3. 分析 AI 输出质量下降的原因
4. 调整提示词模板和上下文
5. 可能需要切换 AI 模型
6. 向用户发送道歉和改进计划
