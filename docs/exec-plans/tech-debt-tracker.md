## 📖 第一章：技术债务概览

### 🎯 什么是技术债务？

技术债务是指在软件开发过程中，为了短期利益而采取的妥协方案，这些方案会在未来产生"利息"，需要额外的努力来偿还。

### 💰 技术债务的类型

1. **设计债务**: 架构缺陷、代码异味
2. **测试债务**: 测试覆盖不足、缺少自动化测试
3. **文档债务**: 文档缺失、文档过时
4. **工具债务**: 构建脚本不完善、CI/CD 缺失
5. **依赖债务**: 过时的库版本、安全漏洞

## 📊 第六章：管理流程

### 1. 发现技术债务

在开发过程中发现需要改进但不紧急修复的问题时，创建新的 TD 文档。

**触发场景**:
- Code Review 时发现代码异味
- 重构需求积累
- 测试覆盖不足
- 文档缺失或过时
- 工具链不完善

### 2. 评估优先级

使用决策矩阵:

```
                    影响范围
                  大        小
              ┌─────────┬─────────┐
            高│ P1      │ P2      │
  紧急度     │ (高优先)│ (中优先)│
              ├─────────┼─────────┤
            低│ P0      │ P3      │
              │(紧急)  │ (低优先)│
              └─────────┴─────────┘
```

**评估维度**:
- **业务影响**: 是否影响用户体验？
- **技术风险**: 是否可能导致系统故障？
- **维护成本**: 是否增加后续开发难度？
- **安全因素**: 是否存在安全隐患？

### 3. 记录文档

复制模板，填写详细信息，确保：
- ✅ 问题描述清晰（一句话概括 + 详细分析）
- ✅ 解决方案可行（含代码示例）
- ✅ 验收标准明确（可验证的清单）
- ✅ 相关资源完整（外部链接、内部文档）

### 4. 定期审查

**审查周期**: 每周审查

**审查内容**:
- 🔍 优先处理 P0/P1 债务（如有）
- 📅 安排 P2 债务到迭代计划
- 🧹 批量清理 P3 债务（月度清理）
- 📊 更新趋势分析和统计数据

**审查会议**:
- 时间：每周一上午 10 点
- 参与：全体开发人员
- 输出：本周偿还计划

### 5. 偿还验证

完成技术债务后：
1. ✅ 更新状态为"✅ 已解决"
2. 📝 更新 [`tech-debt-tracker.md`](file://d:\workspace\opc-harness\docs\exec-plans\tech-debt-tracker.md)
3. 📸 添加验收证据（截图、测试报告等）
4. 🔄 移动到 `completed/` 目录（可选）

### 优先级规则详解

|--------|----------|----------|--------|

graph LR

    C --> F[创建 PR]
    D --> F

    F --> G[Code Review]

    H --> I[更新文档]

## 💡 第九章：最佳实践

### 1. 小步快跑

每次偿还尽量小的改动，避免大规模重构。

**原则**:
- 单一职责：一次只解决一个问题
- 测试保护：确保有测试覆盖
- 渐进式改进：逐步优化，不求一步到位

### 2. 测试覆盖

偿还前确保有测试保护，避免引入新 bug。

**测试策略**:
- 单元测试：验证核心逻辑
- 集成测试：验证模块协作
- 回归测试：确保不影响现有功能

### 3. 代码审查

技术债务也需要 Code Review，确保质量。

**审查要点**:
- 解决方案是否合理？
- 代码质量是否提升？
- 是否有副作用？
- 文档是否更新？

### 4. 文档更新

偿还后及时更新文档，保持信息同步。

**更新内容**:
- 状态标记为"✅ 已解决"
- 添加偿还日期
- 附上测试报告/截图
- 更新追踪索引

### 5. 自动化

能自动检测的尽量自动化，减少人工成本。

**自动化方向**:
- ESLint 规则检测代码异味
- 单元测试覆盖率检查
- CI/CD 自动部署
- 监控告警系统
