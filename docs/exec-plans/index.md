## 🚀 快速开始

### 场景 1: 创建新的执行计划

**何时**: 从 [MVP版本规划](../product-specs/mvp-roadmap.md) 中选择任务后立即创建

**步骤**:

#### 1. 创建文件
在 `docs/exec-plans/active/` 目录创建新文件：
```bash
# 文件名：{TASK_ID}-{TASK_NAME}.md
# 示例：VC-019-测试任务.md
```

#### 2. 填写基本信息
```
# {TASK_ID}: {TASK_NAME} - 执行计划

**状态**: 🔄 进行中  
**优先级**: P0/P1/P2  
**开始日期**: YYYY-MM-DD  
**预计完成**: YYYY-MM-DD  
**负责人**: OPC-HARNESS Team
```

#### 3. 填写目标
```
## 目标

### 核心目标
- [ ] **主要目标 1**: _从 MVP 规划中提取_
- [ ] **主要目标 2**: _支撑功能目标_
- [ ] **技术目标**: _技术难点_

### P0 任务（必须完成）
- [ ] 任务 1: _关键路径任务_
- [ ] 任务 2: _核心功能依赖_

### P1 任务（尽量完成）
- [ ] 任务 1: _优化改进类_
```

#### 4. 保存并确认
- ✅ 文件位于 `active/` 目录
- ✅ 文件名：`{TASK_ID}-{TASK_NAME}.md`
- ✅ 所有占位符已替换

---

### 场景 2: 更新进度

**填写内容**:
```
### Day 1 进度
**完成度**: 80%
- ✅ 已完成：列出完成的任务
- 🔄 进行中：列出正在进行的任务
- ⏸️ 未开始：列出未开始的任务
- 💡 收获与反思：记录今天的学习和解决方案
```

**要点**:
- 如实填写完成度
- 具体列出三类任务状态
- 记录收获和反思

---

### 场景 3: 任务完成归档

**何时**: 任务完成后（harness:check 通过）

**步骤**:

#### 1. 移动文件
```
mv docs/exec-plans/active/{TASK_ID}-{TASK_NAME}.md \
   docs/exec-plans/completed/{TASK_ID}-{TASK_NAME}.md
```

#### 2. 更新状态
```
**状态**: ✅ 已完成  
**完成日期**: YYYY-MM-DD
```

#### 3. 填写质量指标
```
## 📈 质量指标

### Harness Engineering 合规性
- ✅ Health Score: 100/100
- ✅ 测试覆盖率：95%
- ✅ E2E 测试通过率：100%
- ✅ 架构违规数：0

### 时间效率
- 预计工时：_小时数_
- 实际工时：_小时数_
- 效率偏差：_百分比_
```

#### 4. 更新最后信息
```
**最后更新**: YYYY-MM-DD HH:mm:ss
**更新人**: OPC-HARNESS Team  
**版本**: v1.0  
**状态**: ✅ 已完成
```

1. **基本信息** - 状态、优先级、日期等
2. **目标** - 核心目标、P0/P1/P2 任务
3. **阻塞问题** - 问题和解决方案记录
6. **复盘** - 完成后填写的经验教训

8. **相关资源** - 参考文档、任务关联、交付物

## 💡 最佳实践

### ✅ DO (应该做的)
1. **任务选择后立即创建** - 不要跳过计划直接开发
2. **认真填写复盘** - 具体、可操作
3. **如实记录质量指标** - 用于持续改进
4. **保持文档完整性** - 过程记录完整

### ❌ DON'T (不应该做的)
1. **不要跳过执行计划** - 会导致方向偏离
2. **不要在复盘中写套话** - 要具体
3. **不要忘记归档** - 完成后立即移动
4. **不要虚假填写指标** - 如实记录

## ❓ 常见问题

**Q: 执行计划应该多详细？**  
A: 适中即可。核心目标准确，但不过度细化。

**Q: 如果计划赶不上变化？**  
A: 正常！执行计划是活的文档，可以随时调整内容。


**Q: 复盘应该写什么？**  
A: 三部分：①完成情况 ②经验教训（做得好的 + 需改进的）③下一步行动。要具体可操作。
