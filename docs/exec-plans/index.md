# 执行计划使用指南

> 本目录存储所有活跃和已完成的执行计划  
> **核心理念**: 计划驱动执行，复盘促进改进  
> **文档定位**: Sprint 战略 → 执行计划战术 → 交付物成果

---

## 🚀 快速开始

### 场景 1: 创建新的执行计划

**何时**: 从 [Sprint 规划](../sprint-plans/index.md) 中选择任务后立即创建

**步骤**:

#### 1. 创建文件
在 `docs/exec-plans/active/` 目录创建新文件：
```bash
# 文件名：{TASK_ID}-{TASK_NAME}.md
# 示例：VC-019-测试任务.md
```

#### 2. 填写基本信息
````
# {TASK_ID}: {TASK_NAME}

**状态**: 🔄 进行中  
**优先级**: P0/P1/P2  
**任务类型**: Feature/Infra/Bugfix/TechDebt  
**开始日期**: YYYY-MM-DD  
**预计完成**: YYYY-MM-DD  
**负责人**: OPC-HARNESS Team  
**关联需求**: _关联的 Sprint 任务 ID_
```

#### 3. 填写任务概述
```
## 📋 任务概述

### 背景
_为什么需要这个功能？解决什么问题？_

### 目标
从 Sprint 规划中拆解的核心目标：
- [ ] **业务目标**: _用户价值_
- [ ] **功能目标**: _功能范围_
- [ ] **技术目标**: _技术难点/架构改进_

### 范围
明确包含和不包含的内容：
- ✅ **In Scope**: _包含的功能模块_
- ❌ **Out of Scope**: _不包含的内容_

### 关键结果 (Key Results)
可量化的成功标准：
- [ ] KR1: _例如：Health Score 100/100_
- [ ] KR2: _例如：测试覆盖率 >95%_
- [ ] KR3: _例如：E2E 测试通过率 100%_
```

#### 4. 设计解决方案
```
## 💡 解决方案设计

### 架构设计
_系统架构图、组件关系、数据流等_

### 核心接口/API
_主要的函数签名、接口定义_

### 数据结构
_关键的数据模型、类型定义_

### 技术选型
_使用的技术栈、框架、库及选型理由_
```

#### 5. 保存并确认
- ✅ 文件位于 `active/` 目录
- ✅ 文件名：`{TASK_ID}-{TASK_NAME}.md`
- ✅ 所有占位符已替换
- ✅ 任务概述清晰完整

---

### 场景 2: 执行与进度更新

**何时**: 每日站会前或工作结束时更新

**填写内容**:
```
## 📅 执行日志

### Day 1 (YYYY-MM-DD)
**完成度**: 80%

#### ✅ 已完成
- [具体任务 1] - _简要说明_
- [具体任务 2] - _简要说明_

#### 🔄 进行中
- [具体任务 3] - _当前进度百分比_
- [具体任务 4] - _阻塞情况说明_

#### ⏸️ 未开始
- [具体任务 5] - _计划开始时间_

#### 💡 收获与反思
- **技术收获**: _学到的新技术/技巧_
- **问题与解决**: _遇到的问题及解决方案_
- **改进建议**: _流程/工具/方法的改进建议_

#### 📊 今日指标
- 代码行数：+XXX / -XXX
- 单元测试：新增 X 个
- 文档更新：X 处
```

**要点**:
- 如实填写完成度（0-100%）
- 具体列出三类任务状态
- 记录技术收获和反思
- 量化工作产出

### 场景 3: 阻塞问题处理

**何时**: 遇到无法独立解决的阻塞时

**填写内容**:
```
## 🚧 阻塞问题

### 问题 1: [简短描述]
- **严重程度**: 🔴 高 / 🟡 中 / 🟢 低
- **发现时间**: YYYY-MM-DD HH:mm
- **影响范围**: _影响哪些功能模块_
- **根本原因**: _问题的根本原因分析_
- **尝试方案**: _已尝试的解决方案_
- **所需帮助**: _需要什么资源/人员协助_
- **解决方案**: _最终解决方案（解决后填写）_
- **解决时间**: _问题解决时间_
```

### 场景 4: 任务完成归档

**何时**: 任务完成后（harness:check 通过）

**步骤**:

#### 1. 移动文件
```bash
mv docs/exec-plans/active/{TASK_ID}-{TASK_NAME}.md \
   docs/exec-plans/completed/{TASK_ID}-{TASK_NAME}.md
```

#### 2. 更新文档状态
```

**状态**: ✅ 已完成  
**完成日期**: YYYY-MM-DD  
**总耗时**: X 天  
**负责人**: OPC-HARNESS Team
```

## 📊 执行计划完整结构

一个完整的执行计划包含以下部分：

### 启动阶段
1. **基本信息** - 状态、优先级、日期、类型
2. **任务概述** - 背景、目标、范围、关键结果
3. **解决方案** - 架构设计、接口定义、数据结构

### 执行阶段
4. **执行日志** - 每日进度、收获反思
5. **阻塞问题** - 问题记录和解决方案

### 收尾阶段
6. **交付物** - 代码、测试、文档清单
7. **质量指标** - 量化质量评估
8. **技术亮点** - 创新和最佳实践
9. **复盘总结** - 经验教训和改进计划

### 参考资料
10. **相关资源** - 参考文档、任务关联

## 💡 最佳实践

### ✅ DO (应该做的)
1. **任务选择后立即创建** - 不要跳过计划直接开发
2. **认真填写复盘** - 具体、可操作、有数据支撑
3. **如实记录质量指标** - 用于持续改进
4. **保持文档完整性** - 过程记录完整
5. **每日更新进度** - 保持信息同步
6. **及时记录阻塞** - 快速暴露问题
7. **量化工作成果** - 用数据说话

### ❌ DON'T (不应该做的)
1. **不要跳过执行计划** - 会导致方向偏离
2. **不要在复盘中写套话** - 要具体、可操作
3. **不要忘记归档** - 完成后立即移动
4. **不要虚假填写指标** - 如实记录
5. **不要事后补文档** - 边做边记录
6. **不要只报喜不报忧** - 问题要及时暴露

```
graph LR
    A[从 MVP 选择任务] --> B[创建执行计划<br/>active/]
    B --> C[填写任务概述]
    C --> D[设计解决方案]
    D --> E[执行任务<br/>每日更新日志]
    E --> F{遇到问题？}
    F -->|是 | G[记录阻塞问题]

    H -->|是 | E
    F -->|否 | E
    E --> J{完成？}
    J -->|否 | E
    J -->|是 | K[harness:check 验证]
    K --> L{通过？}
    L -->|否 | M[修复问题]
    M --> K
    L -->|是 | N[归档到 completed/]
    N --> O[填写交付物]
    O --> P[填写质量指标]
    P --> Q[填写技术亮点]
    Q --> R[填写复盘总结]
    R --> S[Git 提交]
