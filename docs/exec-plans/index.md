## 🚀 快速开始

### 场景 1: 创建新的执行计划

**何时**: 从 [MVP版本规划](../product-specs/mvp-roadmap.md) 中选择任务后立即创建

**步骤**:

#### 1. 创建文件
在 `docs/exec-plans/active/` 目录创建新文件：
```bash
# 文件名：{TASK_ID}-执行计划.md
# 示例：VC-019-执行计划.md
```

#### 2. 填写基本信息
```
# {TASK_ID}: {TASK_NAME} - 执行计划

**状态**: 🔄 进行中  
**优先级**: P0/P1/P2  
**开始日期**: YYYY-MM-DD  
**预计完成**: YYYY-MM-DD  
**负责人**: OPC-HARNESS Team
```

#### 3. 填写本周目标
```
## 📋 本周目标

### 核心目标
- [ ] **主要目标 1**: _从 MVP 规划中提取_
- [ ] **主要目标 2**: _支撑功能目标_
- [ ] **技术目标**: _技术难点_

### P0 任务（必须完成）
- [ ] 任务 1: _关键路径任务_
- [ ] 任务 2: _核心功能依赖_

### P1 任务（尽量完成）
- [ ] 任务 1: _优化改进类_
```

#### 4. 调整每日计划
根据任务复杂度调整天数：
- **简单任务（1-2 天）**: 压缩 Day 2
- **中等任务（3-5 天）**: 标准 3 天计划
- **复杂任务（5-7 天）**: 扩展 Day 2 和 Day 3

**Day 1 示例**:
```
### Day 1 - YYYY-MM-DD
**主题**: 任务分析 + 架构学习 + 测试设计

- [ ] **上午**
  - [ ] 阅读 MVP 规划，确认任务目标
  - [ ] 学习架构约束规则
  - [ ] 设计单元测试用例
  
- [ ] **下午**
  - [ ] 编写测试代码（TDD）
  - [ ] 开始实现核心功能
  
- [ ] **晚上**
  - [ ] 运行 harness:check
  - [ ] 更新进度追踪
```

#### 5. 保存并确认
- ✅ 文件位于 `active/` 目录
- ✅ 文件名：`{TASK_ID}-执行计划.md`
- ✅ 所有占位符已替换
- ✅ 日期连续

---

### 场景 2: 每日更新进度

**何时**: 每天结束时花 5 分钟

**填写内容**:
```
### Day 1 进度
**完成度**: 80%
- ✅ 已完成：列出完成的任务
- 🔄 进行中：列出正在进行的任务
- ⏸️ 未开始：列出未开始的任务
- 💡 收获与反思：记录今天的学习和解决方案
```

**要点**:
- 如实填写完成度
- 具体列出三类任务状态
- 记录收获和反思

### 场景 3: 任务完成归档

**何时**: 任务完成后（harness:check 通过）

**步骤**:

#### 1. 移动文件
```
mv docs/exec-plans/active/{TASK_ID}-执行计划.md \
   docs/exec-plans/completed/{TASK_ID}-执行计划.md
```

#### 2. 更新状态
```
**状态**: ✅ 已完成  
**完成日期**: YYYY-MM-DD
```

#### 3. 填写周末复盘
```
## 📝 周末复盘

### ✅ 完成情况
**总体完成度**: 100%
**完成的任务**: 1, 2, 3...
**未完成的任务**: _如有，说明原因_

### 💡 经验教训
**做得好的**:
1. _具体优点_ + _影响_
2. _具体优点_ + _影响_

**需要改进的**:
1. _改进点_ + _行动_
2. _改进点_ + _行动_

### 🎯 下一步行动
**立即行动**: [ ] 任务 1, [ ] 任务 2
**短期计划**: [ ] 计划 1, [ ] 计划 2
```

#### 4. 填写质量指标
```
## 📈 质量指标

### Harness Engineering 合规性
- ✅ Health Score: 100/100
- ✅ 测试覆盖率：95%
- ✅ E2E 测试通过率：100%
- ✅ 架构违规数：0

### 时间效率
- 预计工时：_小时数_
- 实际工时：_小时数_
- 效率偏差：_百分比_
```

#### 5. 更新最后信息
```
**最后更新**: YYYY-MM-DD HH:mm:ss
**更新人**: OPC-HARNESS Team  
**版本**: v1.0  
**状态**: ✅ 已完成
```

## 📊 执行计划结构（完整参考）

一个完整的执行计划包含以下部分：

1. **基本信息** - 状态、优先级、日期等
2. **本周目标** - 核心目标、P0/P1/P2 任务
3. **每日计划** - 按上午/下午/晚上划分
4. **阻塞问题** - 问题和解决方案记录
5. **进度追踪** - 每日实际完成情况（每日更新）
6. **周末复盘** - 完成后填写的经验教训
7. **质量指标** - Health Score、测试覆盖率等
8. **相关资源** - 参考文档、任务关联、交付物

**参考示例**: [`DEMO-001-测试任务 - 执行计划.md`](./active/DEMO-001-测试任务 - 执行计划.md)

## 💡 最佳实践

### ✅ DO (应该做的)
1. **任务选择后立即创建** - 不要跳过计划直接开发
2. **每日更新进度追踪** - 每天结束时花 5 分钟
3. **认真填写周末复盘** - 具体、可操作
4. **如实记录质量指标** - 用于持续改进
5. **保持文档完整性** - 过程记录完整

### ❌ DON'T (不应该做的)
1. **不要跳过执行计划** - 会导致方向偏离
2. **不要忘记每日更新** - 进度追踪是关键
3. **不要在复盘中写套话** - 要具体
4. **不要忘记归档** - 完成后立即移动
5. **不要虚假填写指标** - 如实记录

## ❓ 常见问题

**Q: 执行计划应该多详细？**  
A: 适中即可。核心目标准确，每日计划具体，但不过度细化。

**Q: 如果计划赶不上变化？**  
A: 正常！执行计划是活的文档，可以随时调整日期和内容。

**Q: 每天都要看执行计划吗？**  
A: 是的！早上花 2 分钟看计划，晚上花 5 分钟更新进度。

**Q: 周末复盘应该写什么？**  
A: 三部分：①完成情况 ②经验教训（做得好的 + 需改进的）③下一步行动。要具体可操作。
