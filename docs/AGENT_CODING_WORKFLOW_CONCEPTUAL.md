## 1. 概述

### 1.1 设计愿景

实现**完全自动化的多智能体并行开发流程**，让多个 AI 智能体能够同时、独立地执行用户故事，每个故事完成后立即合并到主分支，真正实现"一人公司"的愿景。

### 1.2 核心价值

- ✅ **零人工干预**: 从故事选择到代码合并全流程自动化
- ✅ **高效并行**: 基于系统负载动态调整并发数，最大化资源利用
- ✅ **安全可靠**: Worktree 隔离 + 自动测试 + 即时合并机制
- ✅ **可追溯**: 完整的审计日志和 Git 历史记录
- ✅ **极简流程**: 移除批次和选择器，基于时间自动获取当前 Sprint

### 1.3 适用范围

本文档适用于 OPC-HARNESS 项目的 **Vibe Coding** 模块，涵盖：
- 基于时间的 Sprint 自动执行
- 多智能体并行编码任务调度
- 自动化代码生成、测试、审查和即时合并

## 2. 核心架构理念

### 2.1 整体设计理念

```
┌─────────────────────────────────────┐
│   Time-based Sprint Detection       │
│   (根据当前时间自动获取活跃 Sprint)   │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│   Database Task Coordinator         │
│   (数据库驱动的乐观锁任务竞争)        │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│   Agent Pool Manager                │
│   (动态并发的智能体池管理)           │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│   Instant Merge to Main             │
│   (即时合并 + 两步冲突解决策略)      │
└─────────────────────────────────────┘
```

**架构特点**:
- **数据库驱动**: 所有任务状态存储在 SQLite，支持分布式部署
- **乐观锁竞争**: Agent 通过原子操作竞争任务，无需中央队列
- **无状态 Agent**: Agent 本身不维护任务列表，从数据库动态拉取
- **故障自愈**: 锁超时机制自动回收卡住的任务

### 2.2 关键组件

#### 2.2.1 基于时间的 Sprint 加载器

**职责**: 根据当前时间自动获取活跃的 Sprint 及其用户故事

**工作原理**:
- 查询数据库中当前时间范围内的活跃 Sprint
- 获取该 Sprint 下所有 pending 状态的故事
- 按优先级排序后供 Agent 竞争

**特性**:
- **自动检测**: 无需手动触发，根据系统时间自动识别
- **简单直接**: 移除复杂的评分和依赖分析逻辑
- **时间驱动**: Sprint 的生命周期由起止时间决定

#### 2.2.2 数据库任务协调器

**职责**: 通过数据库原子操作实现分布式任务竞争和状态管理

**核心机制**:
- **乐观锁竞争**: 使用原子 SQL 操作确保只有一个 Agent 能锁定同一故事
- **无状态设计**: 无需维护内存队列，所有状态持久化在数据库中
- **自动故障恢复**: 基于时间戳的锁超时检测，自动回收卡住的任务
- **优先级调度**: 高优先级任务优先执行

**关键优势**:
- ✅ **分布式友好**: 支持跨机器部署多个 Agent 实例
- ✅ **容错性强**: Agent 崩溃不影响其他 Agent，锁超时自动恢复
- ✅ **简化架构**: 移除中央队列服务，降低系统复杂度
- ✅ **可追溯**: 所有状态变更都有数据库审计日志

#### 2.2.3 智能体池管理器

**职责**: 管理 Agent 实例的生命周期和资源分配

**核心功能**:
- **动态并发控制**: 根据 CPU/内存使用率自动调整 Agent 数量
- **Worktree 管理**: 创建/销毁/监控持久化工作树
- **健康检查**: 心跳监控、故障检测、自动重启
- **负载均衡**: 监控各 Agent 负载，动态调整任务分配

#### 2.2.4 编码智能体

**职责**: 在专属 Worktree 中执行用户故事的完整开发生命周期

**执行阶段**:
1. **Analyzing**: 解析 PRD，制定技术方案
2. **Coding**: 调用 AI Provider 生成代码
3. **Testing**: 运行单元测试，自动修复失败
4. **Reviewing**: 执行 Lint/TypeScript 检查
5. **Committing**: 自动生成 Commit Message 并提交
6. **Merging**: 立即合并到 main 分支并重置

**特性**:
- **完全自主**: 无需用户确认任何操作
- **即时合并**: 每个故事完成后立即合并到 main
- **错误恢复**: 自动重试、降级、隔离

## 3. 完整执行流程

### 3.1 阶段 1: 基于时间的 Sprint 自动加载

**时间驱动的 Sprint 检测**:
- 系统根据当前时间自动识别活跃的 Sprint
- 查询该 Sprint 下所有待执行的故事
- 按优先级排序后准备分配

**故事状态流转**:
```
pending → in_progress → completed/failed
```

**资源预检**:
- Git 环境是否正常
- AI Provider API Key 是否有效
- 磁盘空间是否充足
- CPU/内存使用率是否在安全范围

### 3.2 阶段 2: Agent Pool 初始化

**确定并发规模**:
基于系统资源动态计算并发数，平衡性能与稳定性。

**创建持久化 Worktrees**:
为每个 Agent 创建独立的 Git 工作树，提供隔离的开发环境。

**启动 Agent 实例**:
为每个 Worktree 启动一个 Coding Agent 进程，注册到 Agent Manager。

### 3.3 阶段 3: 基于数据库乐观锁的任务竞争

**数据库驱动的任务选择**:
每个 Agent 通过**数据库原子操作**直接竞争选择用户故事，无需内存队列。

**任务竞争机制**:
- 使用原子 SQL 操作确保唯一性
- 只有第一个执行成功的 Agent 能更新成功
- 其他 Agent 的竞争返回失败，表示任务已被锁定
- 自动跳过已被锁定的任务，无需重试逻辑

**关键特性**:
- ✅ **原子性**: 数据库事务保证只有一个 Agent 能锁定同一故事
- ✅ **优先级调度**: 确保高优先级任务先执行
- ✅ **锁超时检测**: 自动回收卡住的任务
- ✅ **无阻塞**: 跳过已锁定行，避免等待

**锁超时与任务回收**:
- 默认超时时间: **30 分钟**
- 超时后自动将任务状态重置为 `pending`
- 防止 Agent 崩溃导致任务永久锁定

### 3.4 阶段 4: Agent 自主执行

每个 Agent 在其专属 Worktree 中顺序执行以下步骤：

```
1. Analyzing Phase (10%)
   - 解析 PRD
   - 分析现有代码
   - 制定技术方案

2. Coding Phase (20%-70%)
   - 调用 AI Provider
   - 流式生成代码
   - 写入文件系统

3. Testing Phase (70%-85%)
   - 生成单元测试
   - 运行测试套件
   - 自动修复失败测试

4. Reviewing Phase (85%-95%)
   - 运行 ESLint/Prettier
   - TypeScript 类型检查
   - 自动修复 lint 错误

5. Committing Phase (95%-100%)
   - 生成 Commit Message
   - 提交代码变更

6. Merging Phase (100%) ⭐
   - 获取分布式合并锁
   - 同步 main 分支最新代码到 agent 分支
   - 在 agent 分支上解决冲突
   - 合并 agent 分支到 main
   - 推送到远程
   - 重置 Agent 分支
   - 释放合并锁

7. 标记任务完成，返回 idle 状态
8. 重复上述流程处理下一个任务
```

### 3.5 阶段 5: 自主迭代执行

**循环逻辑**:
- 持续检查是否还有未执行的任务
- 检查是否有空闲 Agent
- Agents 自动拉取任务并执行
- 动态调整并发策略

**动态策略调整**:
根据执行情况实时优化并发数和资源分配。

### 3.6 阶段 6: 完成总结与报告

**生成统计报告**:
- 总览数据（完成数、失败数、故事点）
- 性能指标（平均时间、最快/最慢故事、并行效率）
- 质量指标（测试覆盖率、Lint 错误修复、冲突解决）
- Agent 性能对比
- 失败故事分析
- 改进建议

**通知用户**:
仅在以下情况通知用户：
- Sprint 完全完成
- 有失败的故事需要人工介入
- 执行时间超出预期

## 4. 异常处理与容错

### 4.1 异常分类与处理策略

| 异常类型 | 处理策略 | 影响范围 |
|---------|---------|---------|
| **Agent 崩溃** | 保存现场 → 重启 Agent → 恢复现场 → 继续执行 | 单个故事 |
| **Worktree 损坏** | 删除损坏 worktree → 重新创建 → 任务重回队列 | 单个 Agent |
| **任务执行失败** | 标记为 failed → 记录日志 → 加入"需人工审查"列表 | 单个故事 |
| **AI Provider 故障** | 切换到备用 Provider → 指数退避重试 → 仍失败则标记失败 | 当前任务 |
| **合并锁竞争失败** | 等待重试间隔 → 最多重试 5 次 → 仍失败则需人工介入 | 单个故事 |
| **合并锁超时** | 定时任务自动回收 → 其他 Agent 可重新竞争 | 全局 |
| **合并冲突** | AI 辅助解决 → 保留双版本 + TODO → 继续合并不阻塞 | 单个故事 |
| **编译失败** | 自动回滚 → 释放合并锁 → 标记需人工介入 → 通知用户 | 单个故事 |
| **资源耗尽** | 降低并发数 → 暂停新任务分配 → 等待资源释放 → 通知用户 | 全局 |

### 4.2 重试机制

采用**指数退避策略**，逐步增加重试间隔，避免频繁重试造成系统压力。

### 4.3 故障转移

**Agent 故障转移**:
1. 获取 Agent 当前任务
2. 释放任务锁
3. 保存现场（未提交的变更）
4. 重启 Agent
5. 恢复现场
6. 可选：重新分配同一任务

### 4.4 数据一致性保证

使用**事务性操作**确保数据库更新的原子性，避免部分成功导致的数据不一致。

## 5. 监控与可观测性

### 5.1 实时监控面板

#### 5.1.1 办公室拟人化动画

**映射关系**:
- Agent 工作状态 → 像素角色在办公室不同区域移动
- 气泡显示当前任务和进度
- 状态动画：打字、呼吸、思考、困惑、庆祝

**状态可视化**:
- `running`: 工作区打字动画
- `idle`: 休息区呼吸动画
- `testing`: 思考动画
- `failed`: 调试区困惑动画
- `completed`: 同步区庆祝动画

#### 5.1.2 实时统计数据

- 活跃/空闲/失败的 Agent 数量
- 队列中的任务数
- 已完成/失败的任务数
- 平均故事执行时间
- 当前吞吐量（stories/hour）
- 系统负载（CPU/内存/磁盘）

### 5.2 详细日志系统

**日志分级**: TRACE → DEBUG → INFO → WARN → ERROR

**结构化日志**:
包含时间戳、级别、Agent ID、故事 ID、执行阶段、消息、元数据等字段。

**日志存储与查询**:
- 本地存储：SQLite + 文件日志
- 日志轮转：每 100MB 或每天轮转
- 查询接口：支持按 Agent、故事、级别过滤

### 5.3 告警机制

**告警规则**:
- 连续失败：同一 Agent 失败 >3 次 → High
- 执行超时：单个故事执行 >60min → Medium
- 资源告警：磁盘使用率 >90% → High
- 队列积压：待执行任务 >50 → Low
- 合并失败：故事合并失败 → High

**告警去重**: 设置冷却期，避免重复告警。

### 5.4 历史追溯

**Sprint 执行历史**:
记录每个 Sprint 的执行时长、完成率、成功率、平均时间等指标。

**故事执行详情**:
记录每个故事的执行 Agent、耗时、状态、提交哈希、文件变更、测试结果等。

## 6. 性能优化策略

### 6.1 并发控制优化

**动态并发调整**:
- 高负载时降低并发（CPU >80% 或内存 >85%）
- 低负载时提高并发（CPU <50% 且内存 <60%）
- 保持合理的并发数以平衡性能和稳定性

**任务预取**:
Agent 在完成当前任务前 10% 时预取下一个任务，减少等待时间。

### 6.2 缓存优化

**AI 响应缓存**:
缓存相似的 AI 请求结果，避免重复调用。

**依赖安装缓存**:
共享 node_modules 缓存，减少重复安装时间。

### 6.3 I/O 优化

**异步文件操作**:
并行写入多个文件，减少 I/O 等待时间。

**Git 操作批量化**:
批量添加文件，减少 git add 调用次数。

### 6.4 网络优化

**AI Provider 连接池**:
复用 AI Provider 连接，减少连接建立开销。

**流式响应处理**:
流式接收 AI 响应，边接收边写入文件，减少内存占用。

## 7. 安全与权限控制

### 7.1 沙箱执行

**文件系统限制**:
Agent 只能访问其专属 worktree，禁止访问项目根目录外的文件。

**禁止危险操作**:
黑名单机制，禁止执行 rm -rf /、sudo、chmod 777 等危险命令。

### 7.2 权限控制

**Agent 权限矩阵**:
- ✅ 读取/写入项目文件（仅限 worktree 内）
- ✅ 执行 Git 命令（仅限当前 worktree）
- ✅ 调用 AI API（使用配置的 API Key）
- ✅ 运行测试（npm test, cargo test 等）
- ❌ 修改配置文件（禁止修改 .env, config.json 等）
- ⚠️ 访问网络（仅限 AI Provider 域名）
- ❌ 执行 Shell 脚本（禁止任意命令执行）

**API Key 安全管理**:
从操作系统密钥链读取 API Key，不硬编码在代码中。

### 7.3 审计日志

**操作审计**:
记录所有敏感操作的时间戳、Agent ID、操作类型、资源、结果等信息。

## 8. 附录

### 8.1 Git 分支管理规范

**分支命名规范**:
```
main                          # 主分支
├─ agent-pool/agent-001       # Agent-001 的持久化分支
├─ agent-pool/agent-002       # Agent-002 的持久化分支
├─ agent-pool/agent-003       # Agent-003 的持久化分支
└─ agent-pool/agent-004       # Agent-004 的持久化分支
```

**注意**: Agent 分支始终保持与 main 同步，每个故事完成后立即合并并重置。

**Commit Message 规范**:
```
<type>(us-<ID>): <subject>

<body>

AI Agent: <agent-id> | Story Points: <points> | Sprint: <sprint-id>
```

**合并策略**:
每个故事完成后立即合并到 main 分支，然后重置 Agent 分支。

### 8.2 配置要点

**Agent Pool 配置**:
- 最大/最小并发数
- 资源阈值（CPU/内存/磁盘）
- Worktree 基础路径和分支前缀
- 锁超时时间和重试策略
- 合并锁配置（超时、重试、回收间隔）

**AI Provider 配置**:
- 主用和备用 Provider
- 模型选择和参数配置
- API Key 环境变量
- 超时和重试策略

### 8.3 常见问题（FAQ）

#### Q1: 为什么采用即时合并而非批次合并？

**A**: 
- **降低复杂度**: 避免批次管理的复杂性
- **减少冲突**: 逐个故事合并，冲突更易解决
- **快速反馈**: 每个故事立即可见，便于及时发现问题
- **简化回滚**: 单个故事回滚比批次回滚更简单

#### Q2: 为什么移除 Story Selector？

**A**:
- **时间驱动**: Sprint 由起止时间自动定义，无需复杂的选择逻辑
- **简化流程**: 直接加载当前 Sprint 的所有 pending 故事，按优先级排序
- **减少决策开销**: 避免复杂的评分和依赖分析
- **更透明**: 用户可以清楚看到哪些故事会被执行

#### Q3: 如何保证多个 Agent 不会选择同一个故事？

**A**:
- **数据库乐观锁**: 使用原子 SQL 操作确保唯一性
- **唯一性保证**: 只有第一个执行成功的 Agent 能更新成功
- **无竞争窗口**: 数据库事务确保不会出现两个 Agent 同时锁定同一故事的情况
- **分布式友好**: 即使 Agent 部署在不同机器上，也能通过数据库保证一致性

#### Q4: 如何处理 Agent 间的代码冲突？

**A**:
- **预防**: 通过故事的依赖字段，确保有依赖的故事按顺序执行
- **检测**: 合并时自动检测冲突
- **解决**: 
  - 简单冲突：AI 辅助自动解决
  - 复杂冲突：保留双版本 + TODO 标记，不阻塞流程
  - 严重冲突：自动回滚，通知用户人工介入

#### Q5: Agent 崩溃后如何恢复？

**A**:
1. **锁超时检测**: 定时任务检查超过阈值的锁定任务
2. **自动回收**: 将超时任务状态重置为 `pending`
3. **重新分配**: 其他 Agent 会自动竞争并领取该任务
4. **现场恢复**: 如果 Agent 重启，可尝试恢复未提交的变更

#### Q6: 如何保证代码质量？

**A**:
- **自动化测试**: 每个故事必须通过单元测试
- **Lint 检查**: 强制执行 ESLint/Prettier
- **TypeScript 类型检查**: 确保类型安全
- **质量门禁**: 测试覆盖率 ≥80%，无 lint 错误
- **自动修复**: AI 自动修复常见问题

#### Q7: 是否可以手动干预执行过程？

**A**:
- **紧急停止**: 提供"紧急停止"按钮，立即终止所有 Agent
- **暂停/恢复**: 支持暂停整个 Sprint 执行，稍后恢复
- **人工审查**: 失败的故事可查看日志，手动修复后重新入队
- **配置调整**: 运行时可调整并发数、合并策略等参数

#### Q8: 如果没有活跃的 Sprint 会怎样？

**A**:
- 系统会记录警告日志
- Agent Pool 不会启动
- 用户可以通过管理界面创建新的 Sprint 或调整现有 Sprint 的时间范围

#### Q9: 数据库驱动相比内存队列有什么优势？

**A**:
- **无单点故障**: 不依赖中央 Task Queue 服务
- **天然分布式**: 支持跨机器部署多个 Agent 实例
- **持久化状态**: 所有任务状态存储在数据库，重启不丢失
- **简化架构**: 移除内存队列的同步和序列化复杂度
- **审计追溯**: 完整的数据库历史记录，便于问题排查

#### Q10: 如何确保多个 Agent 不会同时合并代码？

**A**:
- **分布式合并锁**: 使用专门的锁表和原子 SQL 操作
- **互斥保证**: 确保只有一个 Agent 能获取锁
- **自动释放**: 函数退出时自动释放锁
- **超时回收**: 定时任务检查并释放过期的锁
- **重试机制**: 获取锁失败时等待后重试

#### Q11: 为什么采用"先同步 main→agent，再合并 agent→main"的两步策略？

**A**:
- **减少主分支冲突**: 在 agent 分支上先解决与 main 的冲突，避免影响其他 Agent
- **保证线性历史**: main 分支的合并记录更清晰，易于追溯
- **降低回滚风险**: 如果同步失败，只影响当前 Agent，不阻塞全局
- **符合 Git Flow**: 标准的特性分支工作流，业界最佳实践
- **提高成功率**: 提前解决冲突，最终合并到 main 时几乎不会有冲突

#### Q12: 如果 Agent 在合并过程中崩溃会怎样？

**A**:
1. **锁不会永久占用**: 设置超时时间（默认 10 分钟）
2. **自动回收**: 定时任务检查并释放过期的锁
3. **其他 Agent 可继续**: 锁释放后，其他 Agent 可以重新竞争并执行合并
4. **数据一致性**: Git 操作的原子性确保不会出现半完成的合并
