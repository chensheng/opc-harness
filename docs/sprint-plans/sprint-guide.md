## 🏗️ 敏捷开发组织方式

### 层次结构

```
Epic (史诗)
  └── Feature (特性)
       └── User Story (用户故事)
            └── Technical Task (技术任务)
```

### 各层级定义

#### 1. Epic (史诗)

**定义**: 大型战略目标，通常需要多个 Sprint 才能完成

**特点**:
- 跨越时间长（2-4 个 Sprint）
- 包含多个 Feature
- 直接对齐业务目标
- 由 Product Owner 负责

**示例**:
```markdown
## EPIC-02: Vibe Coding - AI 编码

**史诗目标**: 让 AI 自主完成从任务解析到代码实现的全流程，支持多人协作和质量保障
**完成度**: 67% (24/36 任务完成)
**优先级**: P0
```

#### 2. Feature (特性)

**定义**: 可交付的用户价值单元，通常在一个 Sprint 内完成

**特点**:
- 相对独立的功能模块
- 可在一个 Sprint 内完成
- 包含多个用户故事或技术任务
- 有明确的验收标准

**示例**:
```markdown
### Feature-02.4: 质量门禁基础

**特性描述**: 作为系统，我希望提供基础的代码质量保障，以便确保生成的代码符合规范
**优先级**: P0 | **状态**: ✅ 100% 完成
```

#### 3. User Story (用户故事)

**定义**: 从用户角度描述的功能需求

**格式**: 
```
作为 [角色]，
我希望 [功能]，
以便 [价值/收益]
```

**特点**:
- 以用户为中心
- 描述价值而非实现
- 符合 INVEST 原则
  - Independent（独立的）
  - Negotiable（可协商的）
  - Valuable（有价值的）
  - Estimable（可估算的）
  - Small（小的）
  - Testable（可测试的）

**示例**:
```markdown
| User Story | 作为开发者，我希望生成规范的 Git 提交信息，以便版本管理清晰 | VC-026 | P0 | ✅ 完成 |
```

#### 4. Technical Task (技术任务)

**定义**: 支撑用户故事实现的技术工作

**特点**:
- 面向开发团队
- 描述具体技术实现
- 通常不可见给用户
- 为 User Story 服务

**示例**:
```markdown
| Technical | 实现 Tauri Command `generate_prd`，接入 MiniMax API | VD-005 | P0 | ✅ 完成 |
```

## 📊 表格格式规范

### 基本规则

1. **列宽适中**: 每列内容不宜过宽，保持可读性
2. **对齐方式**: 
   - 文本左对齐
   - 数字右对齐
   - 状态居中
3. **表头清晰**: 使用简洁明了的列名
4. **避免合并单元格**: Markdown 表格不支持复杂合并

### Epic 总览表格式

```markdown
| Epic ID | 史诗名称 | Feature 数 | 任务数 | 已完成 | 完成率 | 状态 |
|---------|---------|-----------|--------|--------|--------|------|
| EPIC-01 | Vibe Design | 4 | 26 | 26 | **100%** | ✅ 完成 |
```

### Feature 详情表格式

```markdown
| 类型 | 用户故事 | 任务 ID | 优先级 | 状态 | 验收标准 |
|------|---------|--------|--------|------|---------|
| User Story | 作为用户，我希望... | VD-001 | P0 | ✅ 完成 | UI 完整，交互流畅 |
| Technical | 实现...功能 | VD-005 | P0 | ✅ 完成 | 能生成结构化 PRD 文档 |
```

### 汇总表格式

```markdown
| 模块 | 总任务数 | 已完成 | 进行中 | 待开始 | 完成率 | 状态 |
|------|---------|--------|--------|--------|--------|------|
| **Vibe Design** | 26 | 26 | 0 | 0 | **100%** | ✅ 完成 |
```

## 🎯 状态标识规范

### Emoji 状态符号

| Emoji | 含义 | 使用场景 |
|-------|------|---------|
| ✅ | 已完成 | 任务 100% 完成，验收通过 |
| 🔄 | 进行中 | 任务已开始但未完成 |
| 📋 | 待开始 | 任务已规划但未开始 |
| ❌ | 已取消 | 任务被取消 |
| ⚠️ | 有风险 | 任务可能延期或有问题 |
| 🎉 | 里程碑完成 | 重要节点或 Module 完成 |

### 状态使用示例

```markdown
| 任务 | 状态 | 说明 |
|------|------|------|
| PRD 生成 | ✅ 完成 | 已通过验收 |
| 用户画像 | 🔄 进行中 | 开发中，完成 80% |
| 竞品分析 | 📋 待开始 | 计划下周开始 |
| 多 AI 厂商 | ❌ 已取消 | 调整到 Sprint 2 |
```

## 🔥 优先级定义

### P0 - 紧急且重要（Must Have）

**定义**: 必须在当前 Sprint 完成的核心功能

**特点**:
- 没有这个功能，Sprint 目标无法达成
- 影响产品核心价值
- 资源优先保障

**示例**:
- 核心业务流程实现
- 关键技术突破
- 阻塞其他任务的基础设施

### P1 - 重要但不紧急（Should Have）

**定义**: 应该在当前 Sprint 完成，但可以灵活调整

**特点**:
- 增强产品竞争力
- 提升用户体验
- 可以在特殊情况下延期

**示例**:
- 功能增强
- 性能优化
- 可视化改进

### P2 - 锦上添花（Nice to Have）

**定义**: 如果时间允许多做的功能

**特点**:
- 非必需功能
- 延迟到下个 Sprint 影响不大
- 通常在 Sprint 后期考虑

**示例**:
- UI 美化
- 额外的配置选项
- 文档完善

## 📐 任务分解原则

### SMART 原则

每个任务应该符合 SMART 原则：

- **S**pecific（具体的）: 明确具体要做什么
- **M**easurable（可衡量的）: 有明确的完成标准
- **A**chievable（可实现的）: 在资源和时间内可完成
- **R**elevant（相关的）: 与 Sprint 目标直接相关
- **T**ime-boxed（有时限的）: 有明确的完成时间

### 任务大小控制

**推荐大小**:
- User Story: 0.5 - 2 人天
- Technical Task: 0.5 - 3 人天
- Feature: 3 - 10 人天
- Epic: 跨多个 Sprint

**过大任务处理**:
如果一个任务超过 3 人天，应该考虑拆分：
```
大任务 → 子任务 1 + 子任务 2 + 子任务 3
```

### 依赖关系管理

**原则**:
1. 尽量减少任务间依赖
2. 优先完成被依赖的任务
3. 并行任务要明确接口

**示例**:
```markdown
VC-001 (PRD 解析器) ← VC-003 (深度 AI 解析)
                         ↓
                    VC-004 (智能任务分解)
```

### 任务ID规范

- 类型清晰: US = 用户故事，TT = 技术任务
- 全局连续: 同类型任务在整个项目中 ID 连续
- 跨 Sprint 唯一: 每个任务 ID 全局唯一
- 易于统计: 可快速计算用户故事/技术任务比例
- 便于扩展: 未来可增加 FT/EP/BUG/DOC 等前缀

## ✓ 验收标准编写

### 验收标准格式

**推荐格式**: Given-When-Then

```
Given [前置条件]
When [操作]
Then [预期结果]
```

### 好的验收标准特征

✅ **好的示例**:
```markdown
- [x] 用户能够在 5 分钟内完成产品构思
- [x] 生成的 PRD 包含完整的功能列表和用户故事（至少 10 个功能点）
- [x] 用户画像具有代表性和差异化（至少 3 个不同画像）
- [x] 竞品分析包含至少 3 个竞争对手的 SWOT 分析
- [x] API 调用成功率 > 95%
- [x] 响应时间 < 3 秒
```

❌ **不好的示例**:
```markdown
- [ ] 实现 PRD 生成功能 （太模糊）
- [ ] 做好用户界面 （不可衡量）
- [ ] 优化性能 （没有具体指标）
- [ ] 修复 Bug （不明确是哪个 Bug）
```

### 不同类型任务的验收标准

#### User Story 验收标准

关注用户价值和体验：
```markdown
- [x] 用户能够在 X 时间内完成 Y 操作
- [x] 界面友好，新手无需培训即可使用
- [x] 提供清晰的错误提示和恢复指引
```

#### Technical Task 验收标准

关注技术指标和质量：
```markdown
- [x] API 响应时间 < 100ms
- [x] 单元测试覆盖率 > 90%
- [x] 无 Critical/High 级别安全漏洞
- [x] 支持并发用户数 > 100
```

## 🌟 最佳实践

### 1. 文档维护

✅ **应该做的**:
- 每周更新文档至少一次
- 及时标记任务状态变化
- 记录重要的决策和变更
- 保持文档与实际进度一致

❌ **不应该做的**:
- 文档写完就不管了
- 状态更新滞后于实际进度
- 缺少变更记录

### 2. 任务粒度

✅ **推荐**:
- 每个任务 0.5-2 人天
- 每个 Feature 3-10 人天
- 每个 Epic 跨 2-4 个 Sprint

❌ **避免**:
- 单个任务超过 5 人天
- 单个 Feature 超过 20 人天
- Epic 边界不清晰

### 3. 沟通协作

✅ **推荐**:
- 每日站会同步进度
- 阻塞问题立即上报
- 定期 Demo 展示成果
- Sprint 结束进行回顾

❌ **避免**:
- 闷头开发不沟通
- 问题积压不及时解决
- 缺少演示和反馈

### 4. 风险管理

✅ **推荐**:
- 提前识别高风险任务
- 准备备选方案（Plan B）
- 定期评估风险状态
- 高风险任务优先处理

❌ **避免**:
- 忽视潜在风险
- 等到最后一刻才处理
- 没有应急预案

### 5. 质量保障

✅ **推荐**:
- 每个任务都有验收标准
- 代码审查必须执行
- 自动化测试覆盖核心功能
- 性能指标量化

❌ **避免**:
- 验收标准模糊
- 跳过代码审查
- 只靠手工测试
- 没有性能基线
